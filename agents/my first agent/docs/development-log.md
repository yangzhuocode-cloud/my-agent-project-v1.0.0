## 2026-03-20

### 上下文管理策略讨论

**背景**：
- 当前使用简单的滑动窗口策略（直接删除旧对话）
- 需要升级为适合工程化场景的智能裁剪策略
- Agent 目标是自动迭代执行代码，需要保持任务连贯性

**调研的裁剪策略**：

1. **滑动窗口策略**（当前方案）
   - 优点：简单高效
   - 缺点：可能丢失重要的早期上下文
   - 适用场景：聊天对话

2. **重要性评分裁剪**
   - 根据消息重要性保留
   - 优先级：System prompt > 最近对话 > 关键信息 > 长回复
   - 适用场景：知识问答

3. **摘要压缩策略**
   - 将旧对话总结成摘要
   - 优点：保留语义，减少 token
   - 缺点：需要额外 API 调用
   - 适用场景：长文档处理

4. **任务导向的分层上下文管理**（推荐方案）
   - 结构：`[System] + [任务描述] + [工具历史摘要] + [最近执行结果] + [当前状态]`
   - 优点：保持任务连贯性，适合迭代执行
   - 适用场景：工程化 Agent、自动化任务

**技术决策**：
- 采用**任务导向的分层上下文管理**策略
- 理由：
  1. Agent 需要记住任务目标和执行结果，而非对话细节
  2. 适合多次迭代执行代码的场景
  3. 错误信息和文件状态需要特殊处理
  4. 参考了 AutoGPT、LangChain Agent 等成熟方案

**核心设计**：

1. **分层存储架构**
   ```python
   context = {
       "permanent": {      # 永久保留
           "system": "...",
           "task": "..."
       },
       "important": {      # 压缩保留
           "errors": [],
           "milestones": []
       },
       "recent": []        # 滚动窗口
   }
   ```

2. **智能压缩规则**
   - 永久保留：System prompt、原始任务、当前状态
   - 完整保留：最近 3-5 轮执行结果、错误信息、文件修改
   - 摘要保留：更早的历史操作
   - 直接删除：重复的错误尝试、无效操作

3. **优先级系统**
   ```python
   CRITICAL = 10   # 永不删除（System、任务）
   HIGH = 8        # 尽量保留（新错误、文件操作）
   MEDIUM = 5      # 可压缩（成功操作）
   LOW = 2         # 可删除（思考过程）
   ```

4. **重复错误检测**
   - 方法：提取错误类型 + 关键信息，生成签名
   - 示例：`NameError: name 'x'` → 签名 `NameError:x`
   - 相同签名的错误视为重复，降低优先级

5. **裁剪触发时机**
   - 在每次添加消息后检查 token 使用率
   - 达到阈值（如 80%）时触发智能裁剪
   - 裁剪目标：降至 60% 使用率

## 开发笔记

### 关键技术点

1. **Token 估算的准确性**
   - 中文字符 × 1.8
   - 英文单词 × 0.25
   - 需要根据实际情况调整系数

2. **上下文管理的权衡**
   - 保留更多历史 → 更好的连贯性，但更快达到限制
   - 激进裁剪 → 节省 token，但可能丢失重要信息
   - 需要根据使用场景调整策略

3. **错误处理的重要性**
   - API 调用失败时需要清理上下文
   - 避免将失败的用户输入保留在历史中
   - 提供清晰的错误提示

### 遇到的问题

1. **问题**：简单的滑动窗口会丢失重要的早期上下文
   - **影响**：Agent 可能忘记用户的原始任务
   - **解决方案**：采用分层上下文管理（待实现）

2. **问题**：重复的错误尝试占用大量 token
   - **影响**：降低可用的上下文空间
   - **解决方案**：实现重复错误检测和压缩（待实现）

3. **问题**：无法区分重要和不重要的消息
   - **影响**：可能删除关键信息，保留无用信息
   - **解决方案**：实现优先级评分系统（待实现）
