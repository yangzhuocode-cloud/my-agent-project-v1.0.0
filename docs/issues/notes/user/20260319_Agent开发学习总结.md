## 核心知识点

### 1. 大模型的无状态性

**关键理解**：每一次对话，对于模型来说都是没有状态的。

- 大模型本身不会记住之前的对话内容
- 在同一次对话中，之前的对话上下文其实全都是由工具商（如 OpenAI、火山方舟等）那边进行封装，帮我们存储的
- 这意味着：如果自己开发 Agent，上下文管理需要自己实现

**实际影响**：
- 需要手动维护对话历史（messages 数组）
- 需要考虑上下文长度限制
- 需要设计上下文裁剪策略

---

### 2. 大模型的缓存机制

**之前的误解**：以为缓存是不同对话之间的缓存

**正确理解**：缓存是同一次对话中，不同 token 计算的缓存

#### 为什么需要缓存？

大模型的计算特点：
- 输入 100 个 token 时，模型需要：
  1. 先计算第 1 个 token
  2. 再计算第 1、2 个 token
  3. 再计算第 1、2、3 个 token
  4. ...以此类推

- 每次都需要把前面的重新计算一遍
- 如果没有缓存，返回速度会越来越慢

#### 缓存层的作用

模型厂商在大模型之外构建了一个缓存层：
- 记录模型在一次对话中，前面 token 计算的中间结果
- 例如：发送 100 个字（100 个 token），模型可以复用前面已计算的结果
- 大幅提升响应速度

### 3. OpenAI 接口协议标准

**关键理解
**：为什么调用不同模型时，都需要填写 `base_url`、`api_key`、`model` 等配置？

#### OpenAI 接口规范

OpenAI 定义了一套标准接口协议，规范了：
- 接口后缀路径
- 入参格式
- 返回值格式

#### 完整的 OpenAI 兼容接口规范

**请求 URL 构成**：
```
完整 URL = base_url + /v1/chat/completions
```

例如：
- OpenAI：`https://api.openai.com/v1/chat/completions`
- 火山方舟：`https://ark.cn-beijing.volces.com/api/coding/v3/chat/completions`

**请求头（Headers）**：
```json
{
  "Content-Type": "application/json",
  "Authorization": "Bearer {API_KEY}"
}
```

**请求体（Body）核心参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `model` | string | 是 | 模型名称（如 gpt-4、Doubao-Seed-2.0-pro） |
| `messages` | array | 是 | 对话消息数组，包含 role 和 content |
| `temperature` | float | 否 | 随机性控制，范围 0-2，默认 1 |
| `top_p` | float | 否 | 核采样，范围 0-1，默认 1 |
| `max_tokens` | int | 否 | 最大生成 token 数 |
| `stream` | boolean | 否 | 是否流式返回，默认 false |
| `presence_penalty` | float | 否 | 存在惩罚，范围 -2.0 到 2.0 |
| `frequency_penalty` | float | 否 | 频率惩罚，范围 -2.0 到 2.0 |
| `n` | int | 否 | 生成几个回复选项，默认 1 |
| `stop` | string/array | 否 | 停止序列 |
| `user` | string | 否 | 用户标识符 |

**messages 数组格式**：
```json
[
  {
    "role": "system",
    "content": "你是一个专业的AI助手"
  },
  {
    "role": "user",
    "content": "你好"
  },
  {
    "role": "assistant",
    "content": "你好！有什么可以帮助你的吗？"
  }
]
```

**返回值格式（非流式）**：
```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1234567890,
  "model": "gpt-4",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "这是回复内容"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 10,
    "completion_tokens": 20,
    "total_tokens": 30
  }
}
```

### 4. 核心参数详解

#### 4.1 stream（流式返回）

**作用**：控制响应方式

- `stream: true`：流式返回
  - 大模型每生成一个 token，就立即返回
  - 前端可以逐字显示，用户体验更好
  - 类似打字机效果

- `stream: false`：非流式返回
  - 等待全部计算完毕后一次性返回
  - 用户需要等待较长时间

**使用场景**：
- 聊天界面 → 使用流式返回
- 后台批处理 → 使用非流式返回

---

#### 4.2 temperature（温度参数）

**作用**：控制输出的随机性和创造性

**工作原理**：
- 大模型根据上下文预测下一个 token
- 每个可能的 token 都有一个概率分布
- temperature 调整这个概率分布的"陡峭程度"

**取值范围**：0 - 2
- `0`：完全确定性，总是选择概率最高的 token
- `0.7`：平衡创造性和准确性（常用默认值）
- `1.0`：标准随机性
- `2.0`：高度随机，输出更有创造性但可能不连贯

**使用建议**：
- 代码生成、数据提取 → 0.1 - 0.3（需要精确）
- 日常对话 → 0.7 - 0.9（平衡）
- 创意写作、头脑风暴 → 1.0 - 1.5（需要创造性）

---

#### 4.3 top_p（核采样）

**作用**：限制候选 token 的范围

**工作原理**：
- 假设下一个 token 有 10,000 个候选
- 每个候选都有不同的概率
- top_p 按概率从高到低累加，直到达到 p 值，其余候选直接排除

**举例说明**：
```
候选 token 概率分布：
- "的" → 30%
- "了" → 25%
- "是" → 20%
- "在" → 10%
- "有" → 5%
- 其他 9,995 个 → 共 10%

如果 top_p = 0.8：
- 累加：30% + 25% + 20% = 75%（未达到 80%）
- 继续：75% + 10% = 85%（超过 80%）
- 结果：只从前 4 个候选中选择，其他 9,996 个完全不可能出现
```

**取值范围**：0 - 1
- `0.1`：只考虑概率最高的 10% 候选
- `0.9`：考虑概率最高的 90% 候选（常用默认值）
- `1.0`：考虑所有候选

**与 temperature 的区别**：
- `temperature`：调整每个候选的概率值
- `top_p`：直接减少候选池的大小

**使用建议**：
- 一般保持 `top_p = 0.9` 的默认值
- 主要调整 `temperature` 来控制随机性
- 两者同时使用时，先应用 temperature，再应用 top_p

### 5. API 调用的本质

**核心认知**：API 只能接收文字，并返回文字

#### 大模型能做什么？
- 输入：纯文本
- 输出：纯文本
- 仅此而已

#### 所有其他功能都需要自己实现

**需要自己搭建的框架**：
1. 上下文存储和管理
2. 上下文裁剪策略（达到长度上限时如何处理）
3. 多轮对话的状态管理
4. 错误处理和重试机制
5. 流式输出的解析
6. 特殊功能（如联网搜索、调用工具等）

**工具商已封装好的功能**：
- Kiro IDE：
  - 上下文存储
  - 达到上限时压缩上一次的上下文
  - 自动进入新的对话
  - 工具调用（文件操作、命令执行等）

- ChatGPT Web：
  - 对话历史管理
  - 上下文自动裁剪
  - 联网搜索
  - 图片生成等

#### 自己开发 Agent 需要做什么？

**最小可用版本**：
1. 维护 messages 数组（上下文）
2. 调用 API 发送请求
3. 解析返回结果
4. 将回复添加到 messages 数组

**进阶功能**：
1. 上下文长度管理（超出限制时裁剪）
2. 错误处理（API 调用失败、超时等）
3. 流式输出处理
4. 多轮对话的自动触发
5. 工具调用能力（Function Calling）

**好消息**：
- 网上有大量开源框架可以直接使用
- 例如：LangChain、LlamaIndex、Semantic Kernel 等
- 不需要从零开始造轮子

## 实践总结

### 开发 Agent 的核心步骤

1. **选择模型和平台**
   - 确定使用哪个大模型（GPT-4、Claude、豆包等）
   - 获取 API Key 和 Base URL

2. **实现基础对话**
   - 构建 messages 数组
   - 发送 HTTP 请求
   - 解析返回结果

3. **添加上下文管理**
   - 维护对话历史
   - 实现上下文裁剪策略

4. **优化用户体验**
   - 添加流式输出
   - 错误处理和重试
   - 加载状态提示

5. **扩展高级功能**
   - 工具调用
   - 联网搜索
   - 多模态支持

## 延伸思考

1. **如何选择合适的上下文裁剪策略？**
   - 固定窗口：只保留最近 N 轮对话
   - 压缩总结：用 AI 总结历史对话
   - 重要性排序：保留最重要的对话

2. **如何处理超长输入？**
   - 分块处理
   - 使用支持更长上下文的模型
   - 提取关键信息

3. **如何提升响应速度？**
   - 使用流式输出
   - 启用缓存
   - 选择更快的模型

4. **如何降低成本？**
   - 优化 prompt 长度
   - 使用更便宜的模型
   - 实现智能缓存策略
