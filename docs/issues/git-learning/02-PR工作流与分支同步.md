# PR 工作流与分支同步

> 📅 创建时间：2026-03-20  
> 🎯 学习目标：理解 PR 工作流的本质、分支同步的必要性、以及如何把握 PR 的时机

## 背景

在学习 nanobot 项目的 Git 提交历史时，发现该项目从一开始就采用了严格的 PR（Pull Request）工作流：
- 所有功能都在独立分支开发
- 通过 PR 审查后合并到 main
- 主分支只接受合并提交，没有直接提交

这引发了对我们项目如何实施 PR 工作流的思考。

## 核心问题

### 1. 什么是 PR 工作流？

PR 工作流是一种代码协作方式：

```
开发者 → 创建功能分支 → 开发 → 推送 → 创建 PR → 审查 → 合并到主分支
```

**关键特点**：
- 代码不直接提交到主分支
- 通过 PR 进行代码审查
- 保持主分支的稳定性
- 每个功能有清晰的记录

### 2. 为什么需要分支同步？

**场景说明**：

```
时间线：
Day 1: 创建 feature/my-agent 分支（基于 master）
Day 2: master 更新了框架（添加新配置系统）
Day 3: feature/my-agent 继续开发（基于旧框架）
Day 5: 准备合并 feature/my-agent 到 master
```

**问题**：
- feature/my-agent 没有 Day 2 的框架更新
- 可能使用了已废弃的 API
- 合并时会产生冲突

**解决方案**：定期同步

```bash
# 在 feature 分支中
git fetch origin
git merge origin/master
# 或使用项目的同步脚本
.\scripts\sync-master-to-worktrees.ps1
```

### 3. 一个 PR 应该包含多少次提交？

**核心原则**：一个 PR = 一个完整的功能或里程碑

**不是按提交次数，而是按功能完成度**

## 适合我们项目的 PR 策略

### 项目特点

1. **框架还在快速迭代**
   - 改动频繁、零散
   - 需要快速试错

2. **单人开发**
   - 暂时不需要多人协作
   - 但需要养成良好习惯

3. **Agent 是独立功能单元**
   - 每个 agent 是完整的功能
   - 适合用 PR 管理

### 推荐策略：混合模式

#### 主分支（框架开发）：直接提交

```bash
# 框架的小改动、快速迭代
git checkout master
# 修改框架代码...
git add .
git commit -m "feat(framework): 添加配置功能"
git push origin master
```

**原因**：
- 改动零散、频繁
- 不需要审查流程
- 保持开发速度

#### 子分支（Agent 开发）：使用 PR

```bash
# 1. 创建 Agent 分支
git checkout -b feature/translator
git worktree add worktrees/translator feature/translator
git checkout master

# 2. 在 worktree 中开发（随意提交）
cd worktrees/translator
git commit -m "wip: 初步实现"
git commit -m "feat: 完成核心功能"
git commit -m "docs: 添加文档"
git push -u origin feature/translator

# 3. 功能完成后创建 PR
# 在 GitHub 上创建 PR: feature/translator → master
# 标题：feat(agent): 添加翻译 Agent
# 描述：实现了 xxx 功能

# 4. 审查后合并
# 在 GitHub 上点击 "Merge Pull Request"

# 5. 清理
cd ../..
git checkout master
git pull
git worktree remove worktrees/translator
git branch -d feature/translator
```

**原因**：
- Agent 是完整的功能单元
- 适合代码审查和记录
- 方便追溯和管理

## PR 时机的判断标准

### ✅ 适合创建 PR 的时机

1. **功能完整**
   - Agent 可以独立运行
   - 核心功能已实现
   - 基本测试通过

2. **文档齐全**
   - 有使用说明
   - 代码有注释
   - README 已更新

3. **代码稳定**
   - 没有明显的 bug
   - 不会频繁大改
   - 可以给别人用了

4. **达到里程碑**
   - v1.0：基础版本
   - v1.1：添加新功能
   - v2.0：重大升级

### ❌ 不适合创建 PR 的时机

1. **功能未完成**
   - 代码还在快速变化
   - 核心逻辑还没写完
   - 运行会报错

2. **提交太零散**
   - 只改了一行代码
   - 只修了个错别字
   - 还在探索阶段

3. **还在调试**
   - 频繁 commit "fix bug"
   - 还在试错
   - 不确定方向

## 分支同步策略

### 同步时机

#### 高频同步（推荐）

```bash
# 每次 master 有重要更新后立即同步
# 在主目录提交框架更新后
git checkout master
git push origin master
.\scripts\sync-master-to-worktrees.ps1
```

**优点**：
- 冲突少且简单
- 始终基于最新代码
- 问题早发现

#### 低频同步（最低要求）

```bash
# 只在准备合并前同步一次
cd worktrees/my-agent
git fetch origin
git merge origin/master
git push
```

**缺点**：
- 可能积累很多冲突
- 合并前需要大量测试
- 可能发现需要重写部分代码

### 同步方法

#### 方法 1：使用项目脚本（推荐）

```bash
# 在主目录执行
.\scripts\sync-master-to-worktrees.ps1  # Windows
bash scripts/sync-master-to-worktrees.sh  # Linux/Mac
```

**优点**：
- 自动同步所有 worktree
- 统一处理
- 减少遗漏

#### 方法 2：手动同步

```bash
# 在 agent 的 worktree 中
cd worktrees/my-agent

# 使用 merge（保留完整历史）
git fetch origin
git merge origin/master

# 或使用 rebase（保持线性历史）
git fetch origin
git rebase origin/master

# 推送更新
git push
```

## 完整的 PR 工作流程

### 场景：开发一个新 Agent

```bash
# === 第 1 步：创建 Agent 分支 ===
git checkout -b feature/translator
git worktree add worktrees/translator feature/translator
git checkout master

# === 第 2 步：开发 Agent（随意提交）===
cd worktrees/translator
# 开发第一版...
git add .
git commit -m "feat(agent): 添加翻译功能"
git push -u origin feature/translator

# === 第 3 步：框架更新了（在主目录）===
cd ../..  # 回到主目录
git checkout master
# 框架更新...
git add .
git commit -m "feat(framework): 更新配置系统"
git push origin master

# === 第 4 步：同步框架更新到 Agent 分支 ===
.\scripts\sync-master-to-worktrees.ps1

# === 第 5 步：继续开发 Agent ===
cd worktrees/translator
# 继续开发...
git add .
git commit -m "feat(agent): 完善翻译功能"
git push

# === 第 6 步：准备创建 PR ===
# 再次确保是最新的
git fetch origin
git merge origin/master
git push

# === 第 7 步：创建 PR ===
# 在 GitHub 上创建 PR: feature/translator → master
# 标题：feat(agent): 添加翻译 Agent
# 描述：
#   - 实现了中英文互译功能
#   - 支持批量翻译
#   - 添加了使用文档

# === 第 8 步：审查和合并 ===
# 在 GitHub 上审查代码
# 检查：
#   - 代码质量
#   - 功能完整性
#   - 文档是否齐全
# 点击 "Merge Pull Request"

# === 第 9 步：清理 ===
cd ../..
git checkout master
git pull  # 拉取合并后的代码
git worktree remove worktrees/translator
git branch -d feature/translator
```

## PR 的不同模式

### 模式 1：一个 Agent 一个 PR（适合小 Agent）

```bash
# 持续开发，随意提交
commit 1: "wip: 初步实现"
commit 2: "wip: 继续开发"
commit 3: "fix: 修复问题"
commit 4: "feat: 完成核心功能"
commit 5: "docs: 添加文档"

# 功能完成后，创建一个 PR
# PR 包含所有这些提交
# 标题：feat(agent): 添加翻译 Agent
```

**适用场景**：
- Agent 功能简单
- 开发周期短（几天内完成）
- 一次性完成所有功能

### 模式 2：分阶段 PR（适合大 Agent）

```bash
# 阶段 1：基础版本
commit 1-5: 基础功能
# 创建 PR #1: feat(agent): 翻译 Agent v1.0 基础版本
# 合并到 master

# 阶段 2：增强功能
commit 6-10: 添加高级功能
# 创建 PR #2: feat(agent): 翻译 Agent v1.1 添加多语言支持
# 合并到 master

# 阶段 3：优化
commit 11-15: 性能优化
# 创建 PR #3: perf(agent): 翻译 Agent v1.2 性能优化
# 合并到 master
```

**适用场景**：
- Agent 功能复杂
- 开发周期长
- 需要分阶段交付

### 模式 3：Draft PR（边开发边审查）

```bash
# 1. 开始开发就创建 Draft PR
git commit -m "feat: 初步实现"
git push
# 在 GitHub 创建 Draft PR（草稿状态）

# 2. 持续开发，持续推送
git commit -m "feat: 继续开发"
git push
# Draft PR 会自动更新

# 3. 完成后，转为正式 PR
# 在 GitHub 上点击 "Ready for review"
# 审查后合并
```

**适用场景**：
- 需要早期反馈
- 想让别人看到进度
- 复杂功能需要讨论

- 状态：还在开发中
- 提交：多次零散提交

```bash
# 描述：这个 agent 可以做什么，怎么用
# 合并到 master
```

**选项 2：分阶段 PR**

```bash
# 如果 my-first-agent 很复杂，可以分几次 PR

# PR #1: 基础版本（能跑起来）
# PR #2: 完善功能（更好用）
# PR #3: 优化和文档（完美）
```

## 最佳实践总结

### 1. 框架开发

- ✅ 直接在 master 提交
- ✅ 保持快速迭代
- ✅ 小改动不需要 PR

### 2. Agent 开发

- ✅ 使用独立分支 + worktree
- ✅ 开发过程随意提交
- ✅ 功能完成后创建 PR
- ✅ 一个 PR = 一个完整功能

### 3. 分支同步

- ✅ 每次 master 更新后立即同步
- ✅ 使用项目脚本自动同步
- ✅ PR 前最后检查一次

### 4. PR 时机

- ✅ 功能完整、文档齐全、代码稳定
- ❌ 功能未完成、还在调试、提交太零散

### 5. 提交规范

- ✅ 遵循 Conventional Commits
- ✅ 提交信息清晰明确
- ✅ 一个 PR 可以包含多个提交
