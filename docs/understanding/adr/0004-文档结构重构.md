# ADR-0004: 文档结构重构

**状态**: 已采纳  
**日期**: 2026-03-19  
**决策者**: 项目团队

## 背景和问题

项目文档存在以下问题：

1. **内容重复严重**
   - README.md、development-guide.md、.kiro/steering/main.md 包含大量重复内容
   - 同一信息在多处维护，容易不一致

2. **目标读者不清晰**
   - 给人类看的理解性内容和给 AI 看的执行性指令混在一起
   - development-guide.md 既有设计理念又有代码规范

3. **文档层级混乱**
   - docs/ 目录扁平化，难以分类查找
   - 没有明确的文档组织原则

4. **维护困难**
   - 修改一处需要同步多处
   - 不知道该在哪个文档更新信息

## 考虑的方案

### 方案 1: 保持现状

**描述**: 不做任何改动，继续使用现有结构

**优点**:
- 无需改动，成本为零
- 不影响现有使用习惯

**缺点**:
- 问题持续存在
- 随着项目发展会越来越混乱
- 维护成本持续增加

### 方案 2: 人机分离 + 功能分类

**描述**: 
- 人类文档按功能分类（understanding/guides/development/reference）
- AI 指令独立管理（.kiro/instructions/）
- 避免内容重复，通过引用组织

**优点**:
- 职责清晰，每个文档有明确的目标读者
- 避免内容重复，易于维护
- 便于查找和扩展
- 符合"人机分离"的设计理念

**缺点**:
- 需要重构现有文档
- 需要更新所有引用链接
- 短期内有学习成本

### 方案 3: 完全合并

**描述**: 将所有文档合并为少数几个大文件

**优点**:
- 文档数量少
- 查找时不需要跳转

**缺点**:
- 单个文件过大，难以维护
- 目标读者混淆
- 不符合单一职责原则

## 决策

选择方案 2：人机分离 + 功能分类

**理由**:

1. **符合设计理念**
   - 项目本身就强调"人机分离"
   - 文档结构应该体现这一理念

2. **长期可维护性**
   - 虽然短期有重构成本，但长期维护更容易
   - 避免技术债务累积

3. **清晰的职责划分**
   - 人类文档解释"为什么"和"怎么理解"
   - AI 指令定义"怎么做"和"边界规则"

4. **可扩展性**
   - 按功能分类便于后续添加新文档
   - 不会破坏现有结构

## 影响

**正面影响**:

1. **文档质量提升**
   - 每个文档职责清晰
   - 内容不再重复
   - 易于查找和理解

2. **维护成本降低**
   - 修改一处即可
   - 明确知道该更新哪个文档

3. **用户体验改善**
   - 人类开发者能快速找到需要的文档
   - AI 助手有明确的执行指令

**负面影响**:

1. **短期学习成本**
   - 需要熟悉新的文档结构
   - 需要更新书签和引用

2. **迁移工作量**
   - 需要移动和重组现有文档
   - 需要更新所有交叉引用

**需要的工作**:

1. 创建新的目录结构
2. 拆分 development-guide.md
3. 移动现有文档到对应分类
4. 创建 .kiro/instructions/ 目录
5. 更新所有文档间的引用
6. 更新 README.md 和 INDEX.md
7. 创建迁移说明文档
