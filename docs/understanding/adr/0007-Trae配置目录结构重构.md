# ADR-0007: Trae 配置目录结构重构

**状态**: 已采纳  
**日期**: 2026-03-22  
**决策者**: 项目团队

## 背景和问题

原有 `.trae/` 目录结构存在以下问题：

1. **入口分散**：`skills_index_full.md` 是索引、`project_rules.md` 是核心规则、各 `SKILL.md` 是具体实现，缺乏单一统一的入口
2. **规则与 Skill 耦合**：`project_rules.md` 直接引用多个外部文件，Trae 检索时需要多次加载，效率偏低
3. **结构层级不明确**：Rule（规则）和 Skill（技能）的定位模糊
4. **目录扁平**：所有 Skill 在同一层级，无法体现优先级差异

## 考虑的方案

### 方案 1: 保持现有结构

**描述**: 维持现有的扁平结构，通过 `skills_index_full.md` 索引所有 Skill

**优点**:
- 无需迁移，保持稳定

**缺点**:
- 入口分散问题未解决
- 无法体现优先级差异
- 维护成本高

### 方案 2: 单一入口 + 分层 Skill（已采纳）

**描述**: 
- 创建 `rules/index.md` 作为唯一入口
- 将 Skill 按优先级分层：critical/high/normal
- 规则与 Skill 解耦，规则定义边界，技能定义执行细则

**优点**:
- 单一入口，降低检索成本
- 优先级分层，按需加载
- 规则与 Skill 解耦，便于维护
- 扩展性强，新增 Skill 只需添加索引

**缺点**:
- 需要迁移现有文件
- 结构变更需要学习成本

### 方案 3: 完全扁平化

**描述**: 将所有规则和 Skill 合并到单一文件

**优点**:
- 最简单的结构

**缺点**:
- 文件过大，难以维护
- 无法按需加载
- 不符合单一职责原则

## 决策

选择方案 2：单一入口 + 分层 Skill

**理由**:
1. 符合 Trae 的「单一入口索引 + 分层 Skill + 规则内聚」设计理念
2. 优先级分层适配 Trae 的分布式元数据检索特性
3. 规则与 Skill 解耦，符合「单一职责」原则
4. 扩展性强，便于后续维护

## 影响

**正面影响**:
- Trae 检索效率提升
- 维护成本降低
- 新增 Skill 更便捷
- 边界清晰，规则管「约束」，Skill 管「执行」

**负面影响**:
- 需要迁移现有文件
- 团队需要学习新结构

**需要的工作**:
- ✅ 创建新的目录结构（critical/high/normal）
- ✅ 创建 `rules/index.md` 作为唯一入口
- ✅ 创建 `rules/core_rules.md` 收敛核心规则
- ✅ 迁移所有 Skill 到对应优先级目录
- ✅ 删除旧文件（`skills_index_full.md`、`operation_details.md`）
- ✅ 更新 `project_rules.md` 引用新入口
- ✅ 更新 `project-structure.md` 文档
- ✅ 创建 `trae-structure.md` 详细说明文档

## 新目录结构

```
.trae/
├── rules/
│   ├── index.md          # 唯一入口：所有Rule/Skill的总索引
│   ├── core_rules.md     # 全局核心规则
│   └── project_rules.md  # 项目规则入口（Trae自动加载）
└── skills/
    ├── critical/         # 启动必加载
    │   ├── branch-management/
    │   └── core-boundaries/
    ├── high/             # 场景触发加载
    │   ├── coding-standards/
    │   └── workflow/
    └── normal/           # 显式调用加载
        └── logging/
```
