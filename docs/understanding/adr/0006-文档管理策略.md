# ADR 0006: 文档管理策略

## 状态

已接受 (2026-03-20)

## 背景

在学习 nanobot 项目的 PR 工作流后，我们思考了项目中各类文档的管理方式。

项目中有大量的辅助性文档：
- `docs/issues/git-learning/` - Git 学习笔记
- `docs/issues/notes/` - 各种总结笔记
- `docs/issues/problems/` - 遇到的问题记录
- `docs/issues/ideas/` - 想法和规划

这些文档的特点：
- 记录零散、频繁
- 不属于任何特定的 agent
- 是项目的知识积累和元数据
- 需要被所有分支访问

### 核心问题

**是否需要为这些文档创建单独的分支进行管理？**

类似于：
- `docs/learning-notes` 分支专门记录学习笔记
- `docs/problems` 分支专门记录问题
- 定期合并到 master

## 考虑的方案

### 方案 1：直接在 master 提交

```bash
# 在主目录（master 分支）
git checkout master

# 记录学习笔记
vim docs/issues/git-learning/03-新学习.md
git add .
git commit -m "docs(learning): 添加 Git 新学习笔记"
git push

# 记录问题
vim docs/issues/problems/user/新问题.md
git add .
git commit -m "docs(problem): 记录遇到的新问题"
git push
```

**优点**：
- ✅ 简单直接，随时记录
- ✅ 这些文档本来就属于项目框架
- ✅ 不需要额外的分支管理
- ✅ 所有 agent 分支都能看到最新的文档
- ✅ 降低记录门槛，提高记录积极性

**缺点**：
- ⚠️ master 分支会有很多零散的文档提交
- ⚠️ 提交历史可能显得不够"干净"

### 方案 2：创建专门的文档分支

```bash
# 创建文档分支
git checkout -b docs/learning-notes
git worktree add worktrees/docs docs/learning-notes

# 在文档分支记录
cd worktrees/docs
vim docs/issues/git-learning/03-新学习.md
git add .
git commit -m "docs: 添加学习笔记"
git push

# 定期合并到 master
cd ../..
git checkout master
git merge docs/learning-notes
git push
```

**优点**：
- ✅ 文档提交和框架提交分离
- ✅ master 历史更清晰
- ✅ 可以独立管理文档的版本

**缺点**：
- ❌ 增加了管理复杂度
- ❌ 需要频繁合并到 master
- ❌ agent 分支看不到最新文档（除非同步）
- ❌ 增加心理负担，降低记录意愿
- ❌ 需要维护额外的 worktree

### 方案 3：混合方式

```bash
# 小笔记：直接在 master
git checkout master
vim docs/issues/notes/user/小笔记.md
git commit -m "docs: 添加小笔记"
git push

# 大文档：用分支 + PR
git checkout -b docs/comprehensive-guide
# 写一个完整的指南...
git commit -m "docs: 添加完整指南"
git push
# 创建 PR，审查后合并
```

**优点**：
- ✅ 灵活，可以根据情况选择
- ✅ 重要文档可以走 PR 流程

**缺点**：
- ❌ 需要判断哪些用分支、哪些直接提交
- ❌ 增加心智负担
- ❌ 标准不统一

## 决策

**选择方案 1：直接在 master 提交**

所有文档记录（学习笔记、问题记录、想法、总结等）都直接在 master 分支提交，不创建单独的文档分支。

## 理由

### 1. 文档本质上属于项目框架

`docs/issues/` 目录和 `docs/understanding/`、`docs/development/` 性质相同，都是项目的知识库和元数据，应该和框架一起管理。

### 2. 零散记录是正常的

学习笔记、问题记录本来就是零散的、即时的，这是它们的特点，不需要像代码那样严格管理。重要的是记录下来，而不是流程的完美。

### 3. 简化工作流程

- 想到就记，不需要切换分支
- 不需要维护额外的 worktree
- 不需要频繁合并
- 降低记录门槛，提高记录积极性

### 4. 所有分支都能访问

agent 分支在同步 master 时，会自动获取最新的文档，开发时可以随时查阅学习笔记和问题记录。

### 5. 提交规范可以保持历史清晰

通过使用明确的提交前缀，可以在查看历史时轻松区分：

```bash
# 学习笔记
docs(learning): xxx

# 问题记录
docs(problem): xxx

# 想法记录
docs(idea): xxx

# 总结笔记
docs(note): xxx

# 框架文档
docs(framework): xxx
docs(guide): xxx

# 框架代码
feat(framework): xxx
fix(framework): xxx
```

这样即使 master 有很多文档提交，也能保持清晰的分类。

## 实施细节

### 日常操作

```bash
# 在主目录（master 分支）
git checkout master

# 场景 1：学习笔记
vim docs/issues/git-learning/03-新学习.md
git add .
git commit -m "docs(learning): Git 新学习笔记"
git push

# 场景 2：问题记录
vim docs/issues/problems/user/新问题.md
git add .
git commit -m "docs(problem): 记录新问题"
git push

# 场景 3：想法记录
vim docs/issues/ideas/user/新想法.md
git add .
git commit -m "docs(idea): 记录新想法"
git push

# 场景 4：总结笔记
vim docs/issues/notes/user/总结.md
git add .
git commit -m "docs(note): 添加总结笔记"
git push

# 同步到所有 agent 分支
.\scripts\sync-master-to-worktrees.ps1
```

### 提交规范

使用 Conventional Commits 格式，明确的 scope 标识：

| 文档类型 | 提交格式 | 示例 |
|---------|---------|------|
| 学习笔记 | `docs(learning):` | `docs(learning): Git worktree 学习笔记` |
| 问题记录 | `docs(problem):` | `docs(problem): 记录 AI 忽略提示词问题` |
| 想法规划 | `docs(idea):` | `docs(idea): Agent 技术演进规划` |
| 总结笔记 | `docs(note):` | `docs(note): 文档重构总结` |
| 框架文档 | `docs(framework):` | `docs(framework): 更新架构文档` |
| 使用指南 | `docs(guide):` | `docs(guide): 完善快速开始指南` |

### 特殊情况

如果某个文档特别重要或复杂（如完整的项目开发指南），可以考虑使用分支 + PR：

```bash
# 创建文档分支
git checkout -b docs/comprehensive-guide

# 写完整的文档，多次提交，慢慢完善
git commit -m "docs: 添加开发指南框架"
git commit -m "docs: 完善配置章节"
git commit -m "docs: 添加示例代码"

# 完成后创建 PR
git push -u origin docs/comprehensive-guide
# 在 GitHub 创建 PR，审查后合并
```

但这种情况应该很少，大部分时候直接在 master 提交即可。

## 影响

### 正面影响

1. **降低记录门槛**
   - 不需要考虑分支管理
   - 想到就记，提高记录频率
   - 有利于知识积累

2. **简化工作流程**
   - 减少分支切换
   - 减少合并操作
   - 降低心智负担

3. **提高文档可见性**
   - 所有分支都能访问最新文档
   - 开发时可以随时查阅
   - 有利于知识共享

4. **保持一致性**
   - 文档和框架一起演进
   - 版本同步，不会出现文档滞后

### 需要注意

1. **提交规范**
   - 必须使用明确的提交前缀
   - 保持提交信息清晰
   - 方便历史查看和过滤

2. **提交频率**
   - 可以频繁提交，但要有意义
   - 避免无意义的提交（如只改一个标点）
   - 可以积累几个小改动一起提交

3. **文档质量**
   - 虽然流程简化，但文档质量不能降低
   - 保持文档的可读性和完整性
   - 定期回顾和整理

## 与其他决策的关系

### 与 ADR 0004（文档结构重构）的关系

ADR 0004 定义了文档的目录结构，本决策定义了文档的管理方式。两者互补：
- ADR 0004：文档放在哪里（结构）
- ADR 0006：文档怎么提交（流程）

### 与 Git 工作流的关系

本决策是整体 Git 工作流的一部分：
- **框架开发**：直接在 master 提交
- **文档记录**：直接在 master 提交（本决策）
- **Agent 开发**：使用 feature 分支 + PR

形成了清晰的三层管理策略。

## 未来可能的调整

### 如果项目变大

如果项目规模扩大，文档提交过于频繁影响 master 历史，可以考虑：
1. 使用 Git 的 `--grep` 过滤文档提交
2. 创建文档专用的 tag 或 release
3. 重新评估是否需要文档分支

### 如果有多人协作

如果有多人同时记录文档，可能需要：
1. 制定更详细的文档规范
2. 考虑使用 PR 审查重要文档
3. 建立文档审查机制

但目前单人开发阶段，直接提交是最合适的。

## 总结

文档管理的核心原则：**简单、直接、降低门槛**

- ✅ 直接在 master 提交
- ✅ 使用明确的提交前缀
- ✅ 保持文档质量
- ✅ 定期同步到 agent 分支
- ❌ 不创建单独的文档分支
- ❌ 不增加额外的管理复杂度
