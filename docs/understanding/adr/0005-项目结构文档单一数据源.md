# ADR 0005: 项目结构文档单一数据源

## 状态
已接受 (2026-03-20)

## 背景

在实施 [ADR 0004: 文档结构重构](./0004-文档结构重构.md) 后，项目文档结构得到了优化。但在后续维护中发现了新的问题：

### 发现的问题

1. **项目结构信息分散**
   - `README.md` 中有简化版的项目结构树
   - `docs/INDEX.md` 中有文档目录结构树
   - `docs/guides/prompts-guide.md` 中有 Prompts 目录结构树
   - `docs/understanding/project-structure.md` 中有完整的项目结构树

2. **维护成本高**
   - 每次目录结构变更需要同步更新多个文档
   - 容易遗漏某些文档，导致信息不一致
   - 增加了文档维护的心智负担

3. **违背单一数据源原则**
   - 项目本身倡导"单一数据源"原则（Prompts 管理）
   - 但文档层面没有贯彻这一原则
   - 存在理念与实践不一致的问题

### 触发事件

在一次文档更新中，用户发现：
- `README.md` 中的项目结构缺少 `docs/agents/` 和 `docs/issues/` 目录
- 多个文档中的结构描述不一致
- 需要手动同步多处内容

这暴露了分散维护的弊端。

## 决策

实施**项目结构文档单一数据源**原则：

### 核心规则

1. **唯一数据源**
   - `docs/understanding/project-structure.md` 是项目结构的唯一维护点
   - 该文件包含完整的目录树、文件职责说明、扩展规范

2. **其他文档引用规范**
   - 禁止在其他文档中复制粘贴项目结构树形图
   - 使用文字引用指向 `project-structure.md`
   - 可以简要说明相关目录用途，但不展示完整树形结构

3. **例外情况**
   - ADR 或技术文档中，可以展示**局部示例结构**（用于说明具体技术点）
   - 示例结构应该是精简的、针对性的，不是完整的项目结构

### 实施方式

#### 文档修改

修改以下文档，移除重复的项目结构树：

1. **README.md**
   ```markdown
   ## 项目结构
   
   完整的项目结构和目录说明请参考：[项目结构说明](./docs/understanding/project-structure.md)
   ```

2. **docs/INDEX.md**
   ```markdown
   ## 📚 文档结构
   
   项目文档按照功能和目标读者分为不同类别，完整的目录结构和说明请参考：[项目结构说明](./understanding/project-structure.md)
   
   主要文档分类：
   - **agents/** - Agent 使用文档
   - **understanding/** - 理解项目
   - ...
   ```

3. **docs/guides/prompts-guide.md**
   ```markdown
   # Prompts 管理指南
   
   完整的项目结构说明请参考：[项目结构说明](../understanding/project-structure.md)
   
   ## Prompts 目录组织
   
   项目中的 Prompts 分为两个层级：
   - **项目级 Prompts** (`prompts/`) - 单一数据源
   - **Agent 级 Prompts** (`agents/{name}/prompts/`) - Agent 专属
   ```

#### AI 指令更新

在 `.kiro/instructions/workflow.md` 中添加规范：

```markdown
#### 文档中的项目结构引用规范

当需要在文档中提及项目结构时：

**禁止**：
- ❌ 在文档中复制粘贴完整的项目结构树形图
- ❌ 在多个文档中维护相同的目录结构说明
- ❌ 创建简化版的项目结构展示

**正确做法**：
- ✅ 使用文字引用指向 project-structure.md
- ✅ 只说明相关的目录用途，不展示树形结构
- ✅ 在 ADR 中可以展示局部示例结构
```

## 影响

### 正面影响

1. **降低维护成本**
   - 项目结构变更只需更新一个文件
   - 避免遗漏和不一致
   - 减少文档维护的心智负担

2. **提高文档质量**
   - 确保所有文档引用的结构信息都是最新的
   - 避免因信息不一致导致的困惑
   - 提升文档的可维护性

3. **理念一致性**
   - 文档层面贯彻"单一数据源"原则
   - 与项目的 Prompts 管理理念保持一致
   - 增强项目设计的整体性

### 负面影响

1. **需要点击链接**
   - 用户需要点击链接才能看到完整的项目结构
   - 但这是可接受的，因为：
     - 完整结构通常只在初次了解项目时需要
     - 文档中会说明相关目录的用途
     - 减少了文档的视觉噪音

2. **初期调整成本**
   - 需要修改多个现有文档
   - 需要更新 AI 指令
   - 但这是一次性成本

## 后续维护

### 维护规则

1. **项目结构变更时**
   - 只需更新 `docs/understanding/project-structure.md`
   - 其他文档的引用会自动指向最新内容

2. **新增文档时**
   - 遵循 `.kiro/instructions/workflow.md` 中的规范
   - 不要复制项目结构树
   - 使用文字引用

3. **文档审查时**
   - 检查是否有重复的项目结构树
   - 确保引用链接正确
   - 验证引用的文档是否存在

### 监控指标

- 项目结构变更时的文档同步时间（应该显著减少）
- 文档不一致的问题数量（应该降为零）
- 新文档遵循规范的比例（应该达到 100%）

## 经验教训

1. **单一数据源原则应该全面贯彻**
   - 不仅适用于代码和配置
   - 也适用于文档和元数据
   - 需要在项目各个层面保持一致

2. **文档维护成本不容忽视**
   - 分散维护会导致信息不一致
   - 应该在设计阶段就考虑维护成本
   - 自动化和规范化是降低成本的关键

3. **理念与实践的一致性很重要**
   - 项目倡导的原则应该在实践中体现
   - 不一致会降低项目的可信度
   - 需要定期审查和改进
