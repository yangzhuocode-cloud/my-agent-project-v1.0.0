# 设计理念

> 📖 **目标读者**: 人类开发者  
> 🎯 **文档类型**: 理解

本文档解释 AI Agents 项目的核心设计理念和架构决策。

## 项目愿景

构建一个灵活、可扩展的 AI Agent 开发框架，让开发者能够：
- 快速创建和部署 AI Agent
- 复用和共享 Prompts 和配置
- 独立导出 Agent 为完整的可运行包

## 核心设计原则

### 1. 单一数据源 (Single Source of Truth)

**理念**: 共享的内容只维护一份，通过引用机制使用。

**体现**:
- 项目级 Prompts 存放在 `prompts/` 目录
- Agent 通过 `#[[prompt:id]]` 引用，而非复制
- 导出时自动解析引用并复制

**为什么**: 避免多处维护相同内容，减少不一致性。

### 2. 开发与导出分离

**理念**: 开发时保持灵活性，导出时保证独立性。

**体现**:
- 开发时：Agent 引用项目级资源
- 导出后：Agent 包含所有必需资源，完全独立

**为什么**: 开发时便于共享和更新，导出后便于分发和部署。

### 3. 人机分离的文档体系

**理念**: 人类和 AI 需要不同类型的文档。

**体现**:
- `docs/` - 给人类看的理解性文档
- `.kiro/steering/` - 给 AI 看的执行性指令

**为什么**: 
- 人类需要理解"为什么"和背景
- AI 需要明确的"做什么"和边界

### 4. 渐进式开发记录

**理念**: 记录开发过程，沉淀知识。

**体现**:
- `docs/issues/` 记录问题、笔记、想法
- User/AI 分离，命名规范不同
- 支持查重和持续更新

**为什么**: 
- 避免重复犯错
- 积累开发经验
- 便于回溯和学习

## 架构决策记录

### 为什么使用虚拟路径引用?

**决策**: 使用 `#[[prompt:id]]` 而非文件路径引用

**原因**:
1. 路径无关性 - 不依赖具体的文件位置
2. 导出友好 - 引用语法在导出前后保持一致
3. 语义清晰 - 通过 ID 而非路径表达引用关系

### 为什么分离 Agent 和项目级 Prompts?

**决策**: 项目级 Prompts 在 `prompts/`，Agent Prompts 在 `agents/{name}/prompts/`

**原因**:
1. 复用性 - 多个 Agent 共享通用规范
2. 维护性 - 规范更新时只需修改一处
3. 独立性 - Agent 也可以有私有 Prompts

### 为什么需要 Agent 导出功能?

**决策**: 提供脚本将 Agent 导出为独立包

**原因**:
1. 分发便利 - 导出的 Agent 可以直接分享给他人
2. 环境隔离 - 不依赖项目环境即可运行
3. 版本固化 - 导出时的版本固定，不受后续更新影响

## 技术选型

### Python 作为主要语言

**原因**:
- AI/ML 生态丰富
- 简洁易读
- 跨平台支持好

### Markdown 作为文档格式

**原因**:
- 人类可读性强
- Git 友好
- 支持 YAML front matter 元数据

### YAML 作为元数据格式

**原因**:
- 结构化数据
- 人类可读
- 易于解析
